# 클로버 완성 비행 연출 설계

작성일: 2026-07-23

## 배경

선행 기록으로 네 번째 잎을 채우면 클로버가 완성되고 보유 클로버가 1 늘어난다.
현재 이 순간의 연출에는 두 가지 문제가 있다.

1. **완성과 숫자 증가가 시각적으로 이어지지 않는다.** 중앙 클로버가 축하 연출을
   재생한 뒤, 우상단 배지(`home_screen.dart:53-69`)의 숫자가 모션 없이 갑자기 바뀐다.
   "내가 만든 클로버가 저기 쌓였다"는 인과가 화면에 드러나지 않는다.
2. **전면광고가 연출을 두 동강 낸다.** `app_controller.dart:175-179`이 완성 1100ms 뒤
   전면광고를 띄우고, 광고가 끝나야 서버가 클로버를 확정한다.

## 목표

완성 → 클로버가 배지로 빨려 올라감 → 배지가 튕기며 숫자 증가를 **한 호흡으로** 잇는다.

## 범위

포함:

- 비행 + 배지 바운스 연출 신규 구현
- 기존 축하 연출에서 링 확산·스파클 제거
- 클로버 완성 시 전면광고 제거

제외:

- 뽑기(클로버 소비) 시 전면광고 추가 — 별도 작업으로 분리한다.
  광고 빈도·스킵 정책·기존 연출과의 충돌을 함께 설계해야 하므로 이번 스펙에 넣지 않는다.
- 뽑기 탭의 보상형 광고(하루 3회 클로버 적립)와 운세 탭 보상형 광고는 그대로 둔다.

## 연출 시퀀스

4번째 잎이 채워진 시점을 0ms로 둔다.

| 시점 | 화면 |
| --- | --- |
| 0ms | 4번째 잎 스프링 팝 (기존 `bounceKey` 경로 유지) |
| ~100ms | 클로버 그룹 전체가 크게 흔들림. **링 확산·스파클 8개 제거** |
| ~700ms | 중앙 클로버가 빈 상태로 리셋. 동시에 완성된 4잎 클로버가 떠올라 호선을 그리며 우상단 배지로 이동하며 축소 (약 500ms) |
| ~1200ms | 배지 도착. 배지 속 클로버가 바운스하고 숫자가 N → N+1 |

전체 약 1.5초.

### 비행 형태

완성된 4잎 클로버가 **통째로 하나** 날아간다. 잎 4장이 각각 흩어져 날아가는 안은
배지 숫자가 1만 오르므로 의미가 어긋나 채택하지 않았다.

### 숫자 지연 규칙

서버는 비행 시작 시점(~700ms)에 이미 증가한 값을 내려준다. 그러나 배지는
**비행이 착지할 때까지 이전 값을 유지**한다. 클로버가 공중에 있는데 숫자가 먼저
올라가면 인과가 깨진다.

## 구조

### 접근 방식

`HomeScreen` 내부 `Stack` + `GlobalKey` 좌표 측정.

`HomeScreen`을 `ConsumerStatefulWidget`으로 전환한다. 중앙 클로버와 우상단 배지에
각각 `GlobalKey`를 달고, `localToGlobal`로 `Stack` 로컬 좌표를 구해 비행 경로를 만든다.

전역 `OverlayEntry`와 투명 `Navigator` 라우트를 검토했으나 채택하지 않았다.
전자는 탭 전환·dispose 타이밍을 수동 관리해야 하고 위젯 테스트가 어렵다. 후자는
`forge_overlay`·`gacha_pull_overlay`와 일관되지만, 그 둘은 사용자가 닫는 모달 연출인
반면 이것은 500ms 스쳐가는 장식 모션이고, 착지 시점의 배지 바운스를 다른 위젯 트리에서
트리거해야 해 조율이 복잡해진다. 시작점과 도착점이 모두 홈 화면 위젯이므로 같은 트리
안에 두는 것이 가장 단순하다.

### 구성 요소

**`CloverCountBadge`** (신규, `lib/widgets/`)
현재 `home_screen.dart:53-69`에 인라인된 배지를 위젯으로 분리한다.
`count`를 받아 표시하고, `count`가 바뀌면 내부 클로버 마크를 바운스시킨다.
언제 새 값을 넘길지는 모른다 — 그 결정은 `HomeScreen`이 한다.

**`CloverFlight`** (신규, `lib/widgets/`)
시작점·도착점·완료 콜백을 받아 호선을 따라 이동하며 축소되는 비행체.
경로 계산과 렌더링만 담당하고 상태를 모른다.

**`HomeScreen`** (수정)
비행 재생 여부와 배지에 넘길 숫자를 결정한다. 표시용 클로버 수를 자체 보유하며,
평소에는 상태를 그대로 따라가되 **비행 중에만 이전 값에 고정**한다.

**`AppState`** (수정)
비행 트리거 신호를 추가한다. `bounceKey`와 같은 증가 카운터 방식을 따른다.
서버 확정이 성공했을 때만 증가한다.

**`app_controller.dart`** (수정)
`_scheduleCloverFinish`의 전면광고 호출을 제거하고 축하 연출 후 바로 확정한다.
확정 성공 시 비행 신호를 증가시킨다.

**`clover_widget.dart`** (수정)
`_CloverPainter.paint`의 링 확산·스파클 렌더링(136-164행)을 제거한다.
그룹 스케일·회전 흔들림은 유지한다.

## 예외 처리

**서버 확정 실패 (오프라인 등).** 비행 신호를 증가시키지 않으므로 비행이 재생되지
않는다. 흔들림까지만 보이고 잎 4개는 그대로 남는다. 다음 실행 시 `_applySnapshot`이
`leaves >= 4`를 감지해 확정 흐름을 재개하고, 그때 비행이 재생된다.

중앙 클로버 리셋을 서버 확정 성공 이후로 묶으므로 **비행을 봤다 = 숫자가 올랐다**가
항상 성립한다.

**비행 중 화면 이탈.** 애니메이션 컨트롤러를 정리하고 표시 숫자를 즉시 최신값으로
맞춘다. 홈으로 돌아오면 이미 증가한 값이 보인다.

**연출 중 재기록.** 진행 중인 비행을 정리하고 새 연출을 시작한다.

**다른 경로의 클로버 증감.** 보상형 광고 적립과 뽑기 소비는 비행 신호를 발생시키지
않는다. 배지는 값 변화에 반응해 바운스하되 비행은 없다.

## 테스트

위젯 테스트 3개를 추가한다.

1. 4잎 완성 시 비행이 재생되고, 완료 후 배지 숫자가 1 증가한다
2. 비행 완료 전에는 배지가 이전 숫자를 유지한다
3. 서버 확정 실패 시 비행이 재생되지 않고 배지 숫자도 변하지 않는다

`app_controller_test.dart:96-98`은 `finishCloverCelebration()`을 직접 호출하므로
광고 제거 후에도 통과한다. `widget_test.dart:108-115`의 `showInterstitial` 테스트는
`AdsController` 자체를 검증하는 것이므로 유지한다 — 호출부는 사라지지만 향후 뽑기
광고에서 다시 쓰인다.

애니메이션의 체감은 테스트로 검증할 수 없다. 구현 후 에뮬레이터에서 직접 실행해
호선 곡률·속도·바운스 세기를 조정한다.
