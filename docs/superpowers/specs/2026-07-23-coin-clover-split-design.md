# 코인·클로버 재화 분리와 커스텀 행운권 설계

작성일: 2026-07-23

## 배경

지금은 클로버 하나가 모든 것을 겸한다. 선행으로 만들고, 광고로 보충하고, 뽑기로
소비한다. `20260713000006_ad_clover.sql`이 "클로버 = 뽑기 코인"이라는 화폐 일원화를
명시적으로 못박은 결과다.

이 문서는 그 결정을 되돌린다. 재화를 둘로 가르고, 선행이 향하는 곳을 뽑기가 아니라
**내 손으로 만드는 행운권**으로 옮긴다.

## 목표

- 뽑기는 코인으로, 커스텀 행운권 제작·강화는 클로버로 한다. 두 재화는 섞이지 않는다.
- 사용자가 문구를 직접 써서 자신만의 행운권을 만들고 키울 수 있게 한다.

## 재화 모델

| | 클로버 | 코인 |
| --- | --- | --- |
| 획득 | 선행 기록 4건(잎 4개) = 1개 | 리워드 광고 1회 = 1개, 하루 5회 |
| 소비 | 커스텀 행운권 제작·강화 | 뽑기 1회 = 1개 |
| 표시 | 홈 우상단 배지 (기존 유지) | 뽑기 탭 상단 카운터 |

두 재화 사이에 환전은 없다. 결제 상품도 두지 않는다.

광고 한도 5회는 `game_config.ad_coins_per_day`로 두어 앱 배포 없이 조절한다.
AdMob에 "N회 초과 시 수익 급감"이라는 공표된 임계값은 없으나, 한 유저에게 리워드
광고를 반복 노출하면 eCPM이 완만히 떨어지고 업계 통용 캡이 5~10회 선이다. 더 큰
이유는 수익이 아니라 리텐션이다 — 한도가 없으면 도감이 하루에 소진된다.

### 감수하는 트레이드오프

뽑기가 선행에서 분리되어 광고에만 묶인다. 선행을 아무리 해도 뽑기는 못 하고, 광고만
보면 뽑기는 할 수 있다. 도감 수집이라는 강한 리텐션 동력이 선행 루프에서 떨어져
나가는 대신, 선행에는 "내 문구로 나만의 행운권을 만든다"는 더 개인적인 보상이 붙는다.
의도된 교환이다.

### 범위에서 제외

**행운권 마켓·유저 간 거래.** 재화 유입이 광고뿐이라 유동성이 없고, 사행성 규제와
미성년자 보호, 분쟁 중재, 어뷰징 계정이 한꺼번에 딸려온다. Supabase RPC 몇 개 규모의
서버가 감당할 기능이 아니다. 더 근본적으로, 마켓이 필요해 보이는 이유는 "강화 재료가
모자라서"인데 재료가 모자란 건 강화를 게임처럼 깊게 만들었기 때문이다. 문제를 스스로
만들고 그 해결책으로 더 큰 시스템을 짓는 순환이다.

**뽑기 행운권의 강화·재조합 규칙 변경.** 지금 로직을 그대로 둔다.

## 커스텀 행운권

사용자가 문구를 직접 쓴 카드다. 카탈로그(`luck_tickets.dart`)에 없다.

- **등급 없음.** 희귀도는 뽑기에서 나온 카드에만 의미가 있다.
- **레벨 있음.** Lv.1~5, `+N` 배지로만 표시된다. 기존 카드와 마찬가지로 레벨은 게임적
  효과가 전혀 없는 순수 과시·애착 장치다(`dex_screen.dart:272-285`).
- **강화는 클로버로, 실패 없음.** Lv.L → L+1에 클로버 L개. Lv.5까지 총 10개.
  등급이 없는 카드에 확률 실패를 붙일 근거가 없고, `+N`의 무게는 실패 리스크가 아니라
  "선행 40건"이라는 시간이 만든다.
- **재료가 되지 않는다.** 재조합·강화 어느 쪽에도 재료로 들어가지 않고, 자신을 강화할
  때 다른 카드를 먹지도 않는다.
- **번역하지 않는다.** 쓴 언어 그대로 저장하고 표시한다.

### 제작 플로우

문구 입력 → 제작 확정 → 리워드 광고 → 카드 공개.

비용은 클로버 1개 + 광고 1회다. 광고가 게이트 역할을 하므로 클로버 비용은 1개로 낮게
잡는다. 광고를 문구 입력 **뒤에** 두는 이유는, 이미 쓴 문구가 있으면 이탈이 적고
광고가 "제작 중"의 뜸들이기로 읽히기 때문이다. 아무것도 쓰기 전에 광고부터 나오면
진입 장벽이 된다.

광고 시청이 실패하거나 중도 이탈하면 **클로버를 차감하지 않는다.** 서버 호출은 광고
완료 콜백 이후에만 일어난다.

강화에는 광고를 붙이지 않는다. 이미 클로버를 여러 개 먹는 데다, 제작에서 한 번 본
유저에게 성장 단계마다 또 요구하면 과하다.

## 서버

### 테이블

`profiles`
- `coins int not null default 0` 추가
- `ad_clovers_today` → `ad_coins_today`
- `last_ad_clover_date` → `last_ad_coin_date`

`custom_tickets` (신규)
- `id uuid pk`, `user_id`, `text text`, `level int not null default 1`, `created_at`
- RLS: 본인 행만 접근

커스텀 카드를 `ticket_instances`에 섞지 않는 이유는 두 가지다. `ticket_catalog` FK가
깨지고, 더 위험하게는 `enhance_ticket`의 재료 개수 검사(`20260713000005`, 74–81행)가
카탈로그 조인 없이 `ticket_instances`만 세기 때문에 커스텀 카드가 재료로 태워질 수
있다. 테이블을 나누면 "커스텀은 재료가 되지 않는다"가 규칙이 아니라 구조로 보장된다.

### RPC

**변경**
- `pull_gacha()` — 클로버 대신 코인 1개 차감. 오류 `NO_CLOVERS` → `NO_COINS`.
  등급 추첨·인스턴스 생성·히스토리 기록은 그대로.
- `grant_ad_clover()` → `grant_ad_coin()` — 코인 1개 지급, 한도는
  `ad_coins_per_day`. 오류 `NO_AD_CLOVERS` → `NO_AD_COINS`.

**신규**
- `create_custom_ticket(p_text text)` — 문구를 트림한 길이가 1~40자인지, 클로버가
  1개 이상인지 검증하고 클로버 1개를 차감한 뒤 카드를 만든다. `history`에 `custom`
  종류로 남긴다. 오류: `INVALID_TEXT`, `NO_CLOVERS`.
- `enhance_custom_ticket(p_id uuid)` — 본인 카드인지, 레벨이 `max_level` 미만인지,
  클로버가 현재 레벨 수 이상인지 검증하고 클로버 L개를 차감해 레벨을 1 올린다.
  성공률 100%. 오류: `TICKET_NOT_OWNED`, `CANNOT_ENHANCE`, `NO_CLOVERS`.

상태 스냅샷 RPC는 `coins`와 커스텀 카드 목록을 함께 내려준다.

### 설정값

`game_config`
- `ad_coins_per_day` = 5 (`ad_clovers_per_day` 대체, 기존 키 삭제)
- `custom_ticket_cost` = 1

강화 비용은 현재 레벨과 같은 값이라 별도 키를 두지 않는다.

### 기존 유저

보유 클로버는 그대로 남고 용도만 바뀐다. 코인은 0에서 시작한다. 클로버를 쌓아두던
유저는 당장 뽑기를 못 하고 광고를 봐야 하지만, 그 클로버로 커스텀 카드를 바로 만들 수
있다. 잃는 것은 없고 쓰임이 이동한다.

`local_game_backend.dart`는 서버 규칙의 거울이므로 같은 규칙을 동시에 반영한다.

## 앱

### 상태와 모델

`AppState`에 `coins`, `adCoinsToday`, `lastAdCoinDate`, `customTickets`를 추가한다.
`clovers`는 그대로 남고 의미만 바뀐다.

`lib/models/custom_ticket.dart` (신규) — id, text, level, createdAt.
`TicketInstance`와 별개 타입이라 실수로 재료 목록에 섞일 수 없다.

`HistoryKind`에 `custom`을 추가해 제작 기록이 아카이브 타임라인에 남는다.

### 뽑기 탭

상단에 코인 카운터와 "광고로 코인 받기 (n/5)" 버튼. 뽑기 버튼은 `coins >= 1`일 때만
활성화되고, 코인이 없으면 광고 버튼으로 시선을 유도한다.

머신에 떨어지는 그림(`gacha_machine.dart:325 _drawCoin`)은 클로버 코인에서 실제
동전으로 바꾼다 — 금색 원반에 가장자리 톱니, 무늬 없음. 클로버 각인을 넣으면 방금
갈라놓은 두 재화가 다시 헷갈린다.

### 지갑 탭

맨 위에 "내가 만든 행운권" 섹션과 "행운권 만들기" 버튼, 그 아래 기존 뽑기 카드 리스트.
재조합·강화 버튼은 지금 자리에 그대로 두되 커스텀 카드는 그 후보 목록에 나타나지 않는다.

커스텀 카드에는 등급색이 없으므로 `rarity_style`이 아닌 전용 스타일(클로버 그린 계열)을
쓰고, 등급 배지 자리에는 `+N`만 표시한다.

`dex_screen.dart`는 이미 헤더·액션 버튼·목록을 한 파일에서 다루고 있어 커스텀 섹션까지
얹으면 무거워진다. `lib/widgets/custom_ticket_card.dart`와
`lib/widgets/custom_section.dart`로 분리하고 `dex_screen`은 조립만 한다.

### 제작 화면

`lib/screens/custom_ticket_create_screen.dart` (신규).

입력 상한 40자. 기존 카드 문구가 한국어 기준 평균 20자, 최대 29자이고 카드가 2줄까지
보여주므로 넉넉하다. 공백만 입력하면 확정 버튼이 비활성이다.

본인만 보고 공유·거래 기능이 없으므로 서버 모더레이션은 두지 않는다.

### 연출

**제작 공개.** 카드가 뒤집히며 등장하고 클로버 배지가 1 줄어드는 것으로 끝낸다.
뽑기는 "무엇이 나올지 모른다"가 재미지만 제작은 이미 답을 아는 행위라 뜸을 들일 이유가
없다. 뽑기 캡슐 연출을 재사용하지 않는다.

**강화.** 커스텀 카드를 탭하면 상세와 "클로버 N개로 강화" 버튼이 나온다. 실패가 없으므로
`forge_overlay`의 게이지·균열·폭발 연출은 쓰지 않고 `+N`이 올라가는 짧은 팝으로 끝낸다.

**홈.** 클로버 배지와 완성 비행 연출은 그대로 둔다.

### l10n

KO/EN/JA 세 언어에 제작·강화·코인 관련 키를 추가한다. 커스텀 카드 문구 자체는 번역
대상이 아니다.

## 테스트

백엔드 규칙:

1. 코인이 0이면 뽑기가 거부된다
2. 광고 한도를 넘으면 코인이 지급되지 않는다
3. 클로버가 0이면 커스텀 제작이 거부된다
4. 문구가 빈 문자열이거나 40자를 넘으면 제작이 거부된다
5. 커스텀 강화 후 레벨이 1 오르고 클로버가 이전 레벨 수만큼 줄어든다
6. Lv.5 카드는 더 강화되지 않는다

위젯:

7. 광고를 완료하지 않으면 클로버가 차감되지 않고 입력 화면으로 돌아온다
8. 커스텀 카드는 재조합·강화 후보 목록에 나타나지 않는다

기존 `app_controller_test.dart`와 `widget_test.dart`에서 클로버를 뽑기 재화로 전제한
부분은 코인 기준으로 고친다.

동전 그림의 곡률·광택과 제작 공개 연출의 속도는 테스트로 검증할 수 없다. 구현 후
에뮬레이터에서 직접 보며 조정한다.
